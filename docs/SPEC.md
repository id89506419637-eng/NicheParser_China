# NicheParser_China — Техническое задание v5.1

## Идея
Автоматический поиск товарных ниш для импорта из Китая в Россию. Агент собирает спрос в РФ (Яндекс Wordstat), предложение в Китае (Alibaba), считает полную себестоимость (ВЭД), маржу на партию и выдаёт вердикт.

## Пользователь
Импортёр/предприниматель, принимающий решения «везти / не везти» товар из Китая.

## Стек
- **Python 3.11+** — бэкенд, пайплайн, парсеры
- **Flask + Jinja2** — веб-интерфейс (dashboard)
- **SQLite** — локальная БД истории
- **Playwright + playwright-stealth** — парсер Alibaba (SPA)
- **Яндекс.Директ API** — Wordstat (получение частотности запросов)
- **OpenRouter (Gemini Flash)** — AI-классификация типа ниши
- **waitress** — WSGI-сервер для Windows (prod-запуск)

## Пайплайн (5 этапов)

### Этап 1 — Анализ спроса (Wordstat)
Источник — Яндекс.Директ Wordstat API. По каждой нише из справочника категорий запрашиваем частотность. Сохраняем в `demand_history` (снимок на текущую дату) для отслеживания динамики.

**Категории (whitelist):**
- Промышленное оборудование и станки
- Медтехника и медоборудование
- Стройматериалы и строительное оборудование
- Сельхозтехника и агрооборудование
- Электроника для бизнеса и производства
- Энергосберегающие технологии и солнечная энергетика
- Датчики, автоматизация, роботизация
- Химия и промышленные материалы
- Инновационные товары (дефицит в РФ)

**Исключить (blacklist):** одежда, обувь, еда, косметика, игрушки, ширпотреб с Ozon/WB.

### Этап 2 — Классификация ниши (AI)
По данным Wordstat (частотность + тренд) и контексту категории AI присваивает тип:
- `ДЕФИЦИТ` — высокий спрос, мало предложения в РФ
- `ИННОВАЦИЯ` — растущий интерес, новый товар
- `СЛАБАЯ НИША` — низкая конкуренция, умеренный спрос
- `ВЫСОКИЙ СПРОС` — массовое потребление
- `ОБЪЁМНЫЙ ТОВАР` — низкая цена единицы, но большие объёмы

Также AI помечает сезонные ниши флагом `is_seasonal`.

### Этап 3 — Поиск на Alibaba (Playwright + stealth)
По каждой нише ищем товары на alibaba.com. Собираем:
- title_en, price_usd_min, price_usd_max
- MOQ, supplier rating, deals_count
- certificates (CE/ISO/RoHS — если есть)
- weight_kg, dimensions_cm (LxWxH)
- product_url
- **конкуренция** — общее число результатов по запросу (для визуализации)

### Этап 4 — Расчёт ВЭД
Для каждого товара прогоняем через `VedCalculator`:
- Закупка (USD × курс ЦБ)
- Пошлина (дефолт 10% — точный код ТН ВЭД уточняет брокер)
- НДС 22%
- Логистика (средний тариф по весу/объёму)
- Банковские комиссии 2%
- **Результат:** `cost_total_rub`, `margin_percent`, `margin_total_rub` (на партию MOQ)

### Этап 5 — Вердикт
- `ВЕЗЁМ` — маржа ≥ 50% И абсолютная маржа на партию ≥ порог (задаётся в settings)
- `ИЗУЧИТЬ` — маржа 30-50% или недостаточная абсолютная сумма
- `НЕ ВЕЗЁМ` — маржа < 30%

## Дашборд (Flask + Jinja2, тёмная тема)

**Главный экран:**
- Статистика: всего ниш / прибыльных / средняя маржа / курс USD
- Таблица топ-10 товаров по марже (название, цена CN, себестоимость RU, маржа %, маржа ₽ на партию, тип ниши, вердикт, ссылка Alibaba)
- Фильтры: категория, вердикт, тип ниши, сезонность, минимальная маржа
- График динамики спроса по неделям (данные из `demand_history`)
- Кнопка «Запустить новый анализ»

**Карточка товара:**
- Все данные Alibaba + разбивка себестоимости по статьям
- Ссылка на Alibaba
- Плашка «Приблизительный расчёт, точный код ТН ВЭД и пошлину уточняет таможенный брокер»

**Настройки:**
- Редактирование параметров ВЭД (пошлина, НДС, логистика, курс, пороги вердикта)

**Модуль Авито:** блок-заглушка «Подключить» — по факту ничего не парсит до появления API-доступа.

## База данных (SQLite)

```sql
niches (id, name_ru, name_en, category, type, is_seasonal, last_frequency, created_at)
products (id, niche_id, title_en, price_usd_min, price_usd_max, moq,
         supplier_rating, deals_count, certificates, weight_kg,
         length_cm, width_cm, height_cm, product_url,
         cost_total_rub, margin_percent, margin_total_rub,
         verdict, competition_count, created_at)
demand_history (id, niche_id, frequency, snapshot_date)
ved_settings (id, usd_rate, cny_rate, duty_percent, vat_percent,
              logistics_per_kg, logistics_per_cbm, bank_percent,
              min_margin_percent, min_margin_total_rub, updated_at)
run_logs (id, started_at, finished_at, status, niches_processed,
          products_found, profitable_count, error_message)
```

## Нефункциональные требования
- Работа на Windows локально (первый запуск), затем опционально Linux-сервер.
- Парсеры изолированы: падение одного не валит пайплайн.
- Случайные задержки 2-5 сек между запросами к внешним ресурсам.
- Ротация User-Agent.
- Логирование в `logs/app.log` с ротацией.
- Секреты (`SECRET_KEY`, `OPENROUTER_API_KEY`, `YANDEX_OAUTH_TOKEN`) только в `.env`.
- Ручная кнопка «запустить анализ» (без крон-джобов в MVP).

## Что в MVP не входит (план на v6.0)
- Авито парсинг (ждём официальный API-доступ)
- Автоматическая классификация кодов ТН ВЭД
- Несколько вариантов доставки (сейчас — один средний тариф)
- Pushing-уведомления о новых прибыльных нишах
- Multi-user / аутентификация
